# AssetWise — Cowork 引き継ぎ文書 (Handoff)

この文書はブレスト結果の確定版です。Cowork はこの文書を唯一の前提とし、「決定済み」は変更せず、「未確定」は実装しながら提案・確定すること。会話ログの断片的な指示よりこの文書を優先する。

## 0. プロジェクト概要

**AssetWise** = ユーザーが身の回りの物を写真に撮るだけで、AIが品目・状態・相場を判定し、**最適な売却チャネルへ振り分ける「全カテゴリ横断リユース・アグリゲーター」**。

ビジョン：家電・家具などの生活用品から、車・不動産まで「売れるものすべて」を1つの入口に集約する。

## 1. 確定した事業設計（変更不可）

### 1-1. コア構造 = 2層アーキテクチャ

プロダクトは以下2層で構成する。**自社で取引機構（オークション・買取・媒介）を最初から持たない**ことが大原則。

**Layer 1（MVP・全カテゴリ入口）= チャネル・ルーター**

- UXは完全ホリゾンタル：何でも写真を撮る → AIが品目/カテゴリ/状態を判定 → カテゴリ別に最適チャネルへ送客。
- 自社で「買い取らない・媒介しない・送客に徹する」。これにより許認可の多くを回避し、不動産・車を含む全カテゴリを初日から載せられる。
- 収益 = 送客アフィリエイト。

| カテゴリ | 送客先チャネル | 収益性 |
| :-- | :-- | :-- |
| 高単価標準品（ガジェット/ブランド/貴金属） | 買取業者ASP送客 | 高（800〜6,600円/件規模） |
| 低単価生活雑貨/家電/家具 | フリマ送客 ＋ まとめ買取送客 | 中〜低 |
| 車 | 既存車一括査定（MOTA/カーセンサー等）へ送客 | 中〜高 |
| 不動産 | 不動産一括査定へ送客 | 高（ただし要法務確認） |

**Layer 2（横展開・内製化）= 自社リバースオークション**

- Layer1で売り手トラフィックと需要データが蓄積した後、高単価・標準カテゴリから MOTA型の自社オークションを内製。
- 買い手の開放順序：**既存業者 → 新規業者開放 → 個人buyer**（個人は最後。理由は §3）。

### 1-2. プロダクトの心臓 = ルーティングテーブル

「カテゴリ → 最適チャネル」の判定が本プロダクトの競争力の中核。データモデルもこれを中心に設計する。

### 1-3. 差別化軸（カテゴリ特化ではない）

既存の「何でも一括査定」アグリゲーター（例：おいくら）は **手入力フォーム＋買取業者のみ**。AssetWise の優位は2点に置く：

1. **写真ゼロ入力のAI査定**（出品情報の生成コストを消す）
2. **フリマ/買取/オークション/車/不動産を横断**するルーティング

※「ブランド品単独」は既存競合多数のため初期の主軸にしない。横断アグリゲーション自体を差別化とする。

## 2. 収益モデル

- **Layer 1**：買取業者への送客アフィリエイト（主収益）。ASP（A8.net / afb / アクセストレード / レントラックス等）経由。
- **Layer 2**：自社オークションの成約手数料 or 業者の月額出展料。
- 補助：プレミアム課金（高精度査定・価格トラッキング・一括査定）。

**収益化不可と確定した項目**：フリマ（メルカリ/ヤフオク/ラクマ/ジモティー）への出品者送客。→ これらは出品無料モデルで出品者集客への報酬プログラムが無い。フリマは「ユーザー価値・集客」目的に限定。

## 3. 規制レーン（[要法務確認]・実装前に境界確定必須）

| 領域 | 関連法 | MVPでの方針 |
| :-- | :-- | :-- |
| 中古品全般の買取 | 古物営業法 | 自社買取しない（送客のみ）→ 回避 |
| 不動産 | 宅地建物取引業法 | 「送客」に徹すれば免許回避の可能性。媒介と判断されれば免許必須 → **要法務確認** |
| 車 | 古物商＋自動車手続 | 既存一括査定へ送客 → 回避 |
| 送客の広告性 | ステマ規制(2023/10〜)・景表法 | 相場提示は中立、送客リンクは「広告/PR」明示で分離 |
| 個人buyer開放 | 古物営業法（反復継続購入） | Layer2 最終段階。C2C（ヤフオク/メルカリ）と正面衝突するため後回し |

注意：当方は法律専門家ではない。収益化・公開前に弁護士確認を前提とすること。

## 4. 技術スタック（確定）

- **Frontend**: Next.js 15 (App Router) / React 19 / TypeScript / Tailwind CSS — `web/` ディレクトリに一本化。
- **Backend**: Python 3.11+ / FastAPI
- **ORM**: SQLAlchemy 2.0 (AsyncSession) + Alembic — 確定。
- **AI/LLM**: Google Gemini（`google-genai` SDK）+ Vision / Structured Outputs（`response_schema`）
- **Database**: PostgreSQL 16（asyncpg ドライバ）
- **Env**: Node.js v20+, Docker

> **フロント構成の確定（2026-05-26 / 会話ログ由来）**: 当初の React Native (Expo) モバイルアプリ案は凍結。プロダクトは `web/`（Next.js 15）の Web アプリ単一に一本化し、旧 `frontend/` ディレクトリ（Expo 由来・Next.js へ転換途中の半端な状態だった）は廃止した。モバイルネイティブ対応は将来フェーズで再評価する。

> **ORM 確定の根拠（旧 [未確定] 論点）**: SQLAlchemy 2.0 を採用。FastAPI の非同期スタック（asyncpg / AsyncSession）と親和し本番採用実績が豊富。Prisma Client Python は非同期サポートと本番採用例が限定的なため不採用。

## 5. 既存の成果物（実装済み・流用前提）

バックエンドの画像解析APIは設計済み。以下の構成を踏襲する。

```
backend/app/
├── main.py          # FastAPI + CORS + lifespan + /health + /api/v1/analyze
├── config.py        # pydantic-settings による設定集約
├── schemas.py       # AnalysisResult / ItemCondition(Enum) 等
└── services/
    └── vision.py    # google-genai (Gemini) + Vision + Structured Outputs
```

**不整合の解消状況（2026-05-26 時点）**：

- [解決済] /analyze の責務分割。`/api/v1/analyze`（品目特定＋ルーティング）と `/api/v1/estimate`（価格帯推定）の2エンドポイントに分割済み。
- [解決済] AI サービスは `services/vision.py` に統一（要件定義の `analyzer.py` は不採用）。プロバイダは Google Gemini。

## 6. 未確定の論点（Cowork が実装しながら確定・提案する）

1. **Layer1 の初期チャネルスコープ**：全カテゴリ送客を一度に実装するか、まず高単価2〜3チャネルに絞るか。
2. **ルーティング判定ロジックの持ち方**：ルールベース（カテゴリ→チャネルの定義テーブル）か、LLM判定か、ハイブリッドか。
3. **不動産・車を MVP に含めるか、Layer1.5 に送るか**：送客先API/提携の有無で実装可否が変わる。提携前は外部リンク送客で暫定対応可。

## 7. Phase 分解（Cowork 投入用）

各タスクは INPUT / DO / DONE / CONSTRAINT 形式。依存が解決した Phase から並列投入可。

| Phase | 名称 | 成果物 | 依存 |
| :-- | :-- | :-- | :-- |
| 1 | ドメイン/データモデル設計 | ルーティングテーブル中心のスキーマ | 本文書 |
| 2 | API契約設計 | OpenAPI仕様・型定義 | P1 |
| 3 | バックエンド実装 | /analyze(査定一括) + ルーター | P2・既存コード |
| 4 | チャネル送客ロジック | 各チャネルへのリンク/送客生成 | P3・§6-1,3確定 |
| 5 | フロント実装 | 撮影→査定→チャネル選択→遷移 | P2 |
| 6 | テスト/CI/デプロイ | Pytest/Jest・Dockerfile | 各Phase |

### 最初のタスク（Phase 1）

```
TASK: ドメイン/データモデル設計（ルーティングテーブル中心）

INPUT: 本ハンドオフ文書（§1-2 が中核）

DO:
- エンティティ定義: Item(品目) / Assessment(査定結果) / Channel(売却チャネル)
  / RoutingRule(カテゴリ→チャネル判定) / AffiliateMeta(ASP・報酬・PR表記フラグ)
- カテゴリ分類体系（高単価標準品 / 低単価生活品 / 車 / 不動産）を定義
- ルーティングテーブルの判定方式（§6-2）を2案比較し推奨を提示

DONE: スキーマが migrate 可能・ER図・選定根拠を README に記載

CONSTRAINT:
- SOLID / DRY 遵守、命名は具体的・意味論的
- 計算量と堅牢性を優先、例外系を網羅
- 査定の中立性と送客の広告性をデータ構造上で分離（ステマ規制対応）
```

## 8. 開発者向け共通制約（userPreferences 由来・全Phase適用）

- 役割：シニアソフトウェアエンジニア兼アーキテクト。簡潔・客観・断定的。
- コード変更前に「影響範囲」と「論理的根拠」を1行で述べる。
- 差分は変更箇所が明確な形式で提示。複雑なロジックに JSDoc/Docstring 付与。
- 既存コード修正時は元のスタイル（インデント等）を厳格に継承。
- セキュリティ（SQLi/XSS/CSRF）はデフォルトで対策。
- 不明な仕様は推測で進めず逆質問。推測実装時は [推測] と明記。
- 廃止・非推奨のライブラリ/メソッドは使用しない。
